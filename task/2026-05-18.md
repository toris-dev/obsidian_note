  핵심 설계 포인트

  1. 파일 충돌 완전 방리

  backend worktree  →  apps/api/ + packages/database/ + packages/trpc/
  frontend worktree →  apps/admin/ + apps/user-app/ + packages/admin-ui/
  두 에이전트가 같은 파일을 절대 건드리지 않도록 소유권을 물리적으로 분리했습니다.

  2. 의존성 순서 강제

  packages/database → apps/api → packages/trpc → apps/admin → apps/user-app
  이 머지 순서를 lead.md · TEAM_WORKFLOW.md 양쪽에 명시해 frontend가 항상 완성된 tRPC 타입을 받아서
  작업합니다.

  3. 통신 프로토콜

  [TASK:backend] / [TASK:frontend]  →  작업 지시
  [DONE:backend] / [DONE:frontend]  →  완료 신호
  [BLOCK:role]                      →  즉시 lead 개입
  [CONFLICT]                        →  git 충돌 중재

  4. Context Drift 방지

  - 각 에이전트 .md 파일에 소유 디렉토리를 명시 → 세션이 길어져도 범위 벗어나지 않음
  - 30분마다 git status · git log 자가 체크 규칙 포함
  - migration은 lead 승인 없이 절대 실행 불가 → 돌이킬 수 없는 변경 보호

  5. 21n 특화 규칙

  - PDF Queue: frontend는 getPdfStatus 폴링만, 실제 Puppeteer는 backend 전담
  - OAuth: EXPO_PUBLIC_* 만 클라이언트 노출 허용, 키는 apps/api/.env.production
  - 전자서명: 상태 전환 로직은 API에만, frontend는 mutation 호출만
  - Migration: prisma-reviewer 에이전트 → lead 승인 2단계 안전망

  최초 1회 수동 진행 순서

  1. expo.dev → Account Settings → Access Tokens → 토큰
  생성
     → GitHub Secrets: EXPO_TOKEN

  2. Google Cloud Console → 서비스 계정 생성 → JSON 키
  발급

  3. Google Play Console → 설정 → API 액세스 → 위 서비스
  계정에 "릴리스 관리자" 권한 부여

  4. JSON 키 파일 전체 내용 → GitHub Secrets:
  GOOGLE_PLAY_SERVICE_ACCOUNT_KEY

  5. eas build --profile production --platform android
  (로컬 1회 실행)
     → EAS가 Android 서명 키스토어를 생성·저장
     → .aab 파일을 Play Console에 수동 업로드 (내부 테스트
   트랙이라도 OK)

  6. 이후부터 main 푸시 시 CD 워크플로우가 자동으로 빌드 +
   제출

  ▎ skip_eas: true 입력으로 workflow_dispatch를 실행하면

  2. Google Cloud Console → 서비스 계정 생성 → JSON 키 발급

  3. Google Play Console → 설정 → API 액세스 → 위 서비스 계정에 "릴리스 관리자" 권한 부여

  4. JSON 키 파일 전체 내용 → GhitHub Secrets: GOOGLE_PLAY_SERVICE_ACCOUNT_KEY
